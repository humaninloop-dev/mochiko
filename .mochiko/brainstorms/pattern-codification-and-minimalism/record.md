#### 2. The as-built pattern (brainstorm.md vs setup.md)

Both team-form commands share, section-for-section:
- **`disable-model-invocation: true`** + a long descriptive frontmatter blurb; a **"You are the lead"** framing; the loop declared as a `mochiko:loop-discipline` sound loop.
- **"Hard requirement — agent teams"** — near-identical prose: env check as a *proxy*, **first teammate spawn is the authoritative probe**, **no fallback transport** (marked `Contested`), transport discriminable at the call (`name:` = seat, spawn without `name:` = forbidden one-shot subagent), **post-spawn addressability check**, recipe pointer to `templates/agent-dispatch.md` (Seat transport). (setup.md:31–46, brainstorm.md:12–14.)
- **"Session constraints"** — never narrate machinery ("phase/round/gate"), never narrate/reply to teammate housekeeping, tell the user they can watch/message any teammate, **announce each seat in one line** ("unexplained spawn reads as a malfunction").
- **"The team"** — teammates **do not load `skills:` frontmatter**, so every spawn brief must name skill + role + inputs (points to `agent-dispatch.md`); reports arrive as messages to a **named standing seat**.
- **Review at convergence** — a user-ruled **sizing gate** (`pair / single / none`-with-recorded-waiver), lens-split pair, reviewers **withheld from each other** until findings form cold, **one-shot four-message cross-examination**, **survivor** routing by answer-owner, a **verify pass**, artifact **frozen** during review.
- **As-you-go artifact with confidence marks** (brainstorm: `record.md` D1… marks `Confident/Assumed/Contested/Unsure/Deferred`; setup: `governance-intent.md` GI-IDs + a confidence mark per element).
- **"Contract (authoring-time fill — governed by `mochiko:loop-discipline`)"** — Done-condition default **FAIL**, Producer↔validator independence, Bounds, Human gates. Both explicitly state **"no per-run contract is written."**
- **"Recovery"** — teams don't survive `/resume`; pause posture + respawn-what-the-stage-needs.
- Cross-exam protocol is **single-sourced**: setup's `review-governance-intent` reuses `review-brainstorm/references/CROSS-EXAM.md`.

## Decisions

- **D1 — Codification target: the artifact shape (A), not the build lifecycle.** The goal-1 primitives codify what `brainstorm.md`/`setup.md` *are* — the team-form architecture (hard team requirement, named seats, human gates, contract section, as-you-go confidence-marked artifact, sized end-stage review, recovery posture) — not *how* they got that way. The build lifecycle (design session → record → build → dogfood → revise) is explicitly out of codification scope: it is the user's own practice — "that build lifecycle happened by me dogfooding and identifying issues." Lead recommended C (shape + lifecycle as one discipline); user ruled A with that reasoning. Lead's one push-back, logged: the shape itself is churn-hot (revised v2→v2.1→v2.2 inside two weeks by exactly that dogfooding), so shape-codifying primitives must be cheap to revise — a constraint goal 2's version-mapping requirement partly covers. `Contested`

- **D2 — Application scope: command-by-command assessment, with team-form as the declared default preference.** No blanket conversion: each of the five one-shot commands gets assessed on its own merits for whether the shape (and which parts of it) fits. But the assessment runs under a stated doctrine, in the user's own words: team-form is preferred because "context is retained over a longer horizon" and lead↔teammate cross-collaboration "tends to create a much better output"; the lead's role is orchestration and interfacing with the human; subagents remain right for *surgical* needs (bounded one-shot lookups/tasks). Lead-noted implication: this demotes `agent-dispatch.md`'s standing line "one-shot dispatch stays correct for the commands designed on it (specify/plan/tasks/implement)" from settled ruling to rebuttable default, pending the per-command assessments. The structural question the lead's Q2-C raised (layered codification: form-agnostic core vs team transport layer) was not ruled on and defers into the vehicle decision. `Confident` *(Fold (S4, user-ruled at review): a team-form conversion requires no pre-existing dogfood evidence — none can exist before the team variant does — but each conversion assessment must record its command-specific rationale, and the first post-conversion dogfood run is a scheduled checkpoint that confirms or reverts the conversion, keeping D2's default coherent with D7's evidence discipline.)*

- **D3 — Vehicle: surface + keeper (Q3-C).** The shape gets one single-sourced **pattern surface** (template and/or skill — the duplicated team-form prose moves there; commands shrink to per-workflow parameters plus references, following the `CROSS-EXAM.md` precedent), plus a **keeper**: an agent + skills with two jobs against that surface — *apply* (author a new command in the shape / convert an existing one, serving D2's per-command conversions) and *audit* (grade an existing command's conformance — refilling the enforcement seat empty since `verify-output`'s altitude gate retired 2026-07-18). Rejected: B alone (goal 1 says "ensures … applied" — an active capability, not documentation) and A alone (buries the shape in a procedure the next dogfood revision can't cheaply reach, violating D1's churn constraint, and audits nothing that already exists). `Confident` *(adoption of lead recommendation, unelaborated — streak count 1; re-affirmed by the user post-review, S9)* *(Refined by D8 (S12): the keeper is structurally two seats. Fold (S2): the template is the shape's **sole authoritative home**, internally layered — form-agnostic core + team-transport layer, closing D2's deferred layering thread; the authoring/audit skills and conformant commands reference it, never restate it, so a shape revision is one template edit + re-audit, honoring D1's churn constraint.)*

- **D4 — Sequencing: codify first, then strip (Q4-A).** Goal 1's surface + keeper get built before the library-wide strip pass runs. Grounds: the dominant strip disposition is expected to be *relocate to the single-sourced home*, not *delete* (the command-altitude lesson — verbosity was mis-altitude, not dead weight), and relocation needs the home to exist; the auditor then checks every strip; single-sourcing the team-form prose already shrinks `brainstorm.md`/`setup.md` as a side effect. Rejected: strip-first (risks deleting lines that should have moved) and one rolling pass (early strips judged against a surface later strips reshape → re-audits). `Confident` *(adoption of lead recommendation, unelaborated — streak count 2; re-affirmed by the user post-review, S9)* *(Fold (S8): the codified home is **provisional** — a home-revision checkpoint runs at the first one-shot→team-form conversion, cheap insurance against a shape need the two founding exemplars didn't reveal.)*

- **D5 — Strip criterion: tiered — altitude test first, purpose-proof second (Q5-C).** Tier 1 (structural, cheap): a line survives only as (i) content unique to the primitive, (ii) a reference to a single-sourced home, or (iii) a marked exception carrying its justification — restated doctrine/pattern is relocated or deleted. Tier 2 (behavioral, on Tier-1 survivors only): the line must name the behavior it produces or the failure it prevents, with provenance where it exists (dogfood defect, session ruling); provenance doubles as the strip log's evidence column and protects hard-won defect-fix lines (e.g. the team-transport addenda) from being stripped as verbosity — the risk the fact-checker's map flags as currently un-mitigated. Rejected: altitude-only (keeps well-placed padding), purpose-only (burns judgment on lines the structural test dispatches mechanically). `Confident` *(adoption of lead recommendation, unelaborated — streak count 3 → streak flagged to user; next fork posed without a marked recommendation; re-affirmed by the user post-review, S9)* *(Folds: (S10) survivor provenance lives in D6's per-primitive note as survivor-provenance entries, not an inline "evidence column" — primitives stay lean; (S13) on large clusters the agent first-passes Tier 2 with per-line rationales and the user ratifies contested/uncertain lines rather than every line — judgment kept, bulk rubber-stamping avoided.)*

- **D6 — Strip log: per-primitive strip notes, non-loaded, version-stamped per entry (Q6-B + two riders).** One file per primitive in a dedicated plugin dir that no loaded surface references (e.g. `plugins/mochiko/strips/<primitive>.md`, same idiom as `templates/`) — *not* siblings inside `commands/` (files there register as commands) and never inside SKILL.md/command bodies (the log must not itself cost context). Each entry: the stripped content, criterion tier it failed (D5), plugin version stamped. Wave summaries stay on the existing chain (ROADMAP/REGISTRY rows) — no new central artifact; C's steelman conceded on that point. Grounds: the operational lookup is per-primitive ("what did we strip from *plan*?" at dogfood time), and the keeper's audit unit (D3) is also the primitive. Posed recommendation-free under the streak flag; user led with B, lead concurred with riders, user confirmed. `Confident` *(streak broken — user-initiated lean)* *(Fold (S10): the per-primitive note carries two entry types — **strip entries** (as ruled) and **survivor-provenance entries** for kept lines citing their Tier-2 evidence; both version-stamped, neither ever inline in the primitive.)*

- **D7 — Re-add protocol: evidence-gated with a marked override (Q7-C), run as a collaborative analysis over dogfood logs.** Default: a stripped line returns only with cited dogfood evidence linked from its strip entry (*returned at vX, evidence: →link*); owner override allowed but marked as such (*returned at vX, override — no linked evidence*), so the log stays honest and override clusters become a hunt-signal for the next audit. The user's stated workflow: they supply dogfooding session logs and work through them with the lead to identify what comes back — re-add candidates are identified against the primitive's strip notes (D6 makes that a per-primitive lookup), then the user rules each re-add. Rejected: strict evidence-only (unenforceable against the owner → silently bent log) and pure judgment (re-bloat by accumulated small judgments — how the verbosity got there originally). `Confident` *(elaborated adoption — user added the dogfood-log workflow)* *(Fold (S5): steady-state cadence — any version bump containing re-adds triggers the validator audit on the touched primitives, and an override-count threshold (build-time parameter) flags a primitive for audit regardless — the honest-log signal always has a consumer.)*

- **D8 — Goal-1 build: new author agent + existing grader + template surface (Q8-B).** A new framework-facing agent (working name `command-architect`) hosts a new authoring skill (build/convert a command to the shape, serving D2's per-command conversions); the audit lands as a new `validation-*` skill on the existing `mochiko:validator` — apply and audit on different agents because no agent grades its own output (this corrects D3's loose "an agent + skills" phrasing; the keeper is structurally two seats). The pattern surface lives as a template beside `workflow-contract.md`/`agent-dispatch.md` — the authoring-time-fill idiom the two live team-form commands already consume. Rejected: no-new-agent (leaves "ensures applied" unowned *(ground superseded — see S6 fold)*; no existing persona is framework-smithing-shaped — `principal-architect` governs projects, not this plugin) and a full keeper team (+2 agents +1 command added to a roster about to be stripped; duplicates `validator`'s grader role). `Confident` *(adoption of lead recommendation, unelaborated — new streak count 1; re-affirmed by the user post-review, S9)* *(Folds: (S6, owner-accepted verbatim) justification corrected — command-architect stands on **delegated authoring-craft** grounds under D2's doctrine (converting a live command is produced craft for a teammate), not on "ensures applied," which the audit + template carry; (S1) the `validation-*` audit skill gains a **deterministic conformance floor** — grep-checkable: references present, no restated doctrine, exceptions marked; `verify-output`'s floor+ceiling design revived — beneath its prose judgment, with the residual prose-layer rationalization risk recorded as accepted; (S2) both new skills **reference** the template, never restate it; (S3) the keeper's own primitives are classed shared primitives under D9's discipline, every authored/converted command is an authoring-skill dogfood datapoint reviewed at that cluster's audit, and the authoring skill's thinner re-add-evidence exposure is a marked known limit.)*

- **D9 — Strip-pass execution: cluster waves, shared primitives handled in-context (user-shaped twice).** One command **plus its related agents and skills** per wave — the user's modification of the lead's class-waves proposal, and the stronger design: it matches CLAUDE.md's port-the-cluster-together doctrine, and dogfood evidence arrives per workflow run, so one wave = one workflow = one version stamp (the D6 mapping). Shared primitives (`loop-discipline`, router, `workflow-contract`/`agent-dispatch`, `validator`, `devils-advocate`) get **no closing wave** (lead's proposal, declined): the user ruled they are modified in-context, during whichever cluster wave surfaces the need. Note: `brainstorm`/`setup` largely pre-shrink during D4's codification (relocation into the pattern surface *is* their strip) — their wave is a light residual pass. `Confident` *(Folds: (S7, user-ruled at review) the in-context rule stands for low-fanout shared primitives; a primitive with **≥3 consumers** has its strip *raised* in-wave but ruled and audited as a scheduled all-consumer escalation, never judged inside one cluster's context alone; (S3) command-architect, both new skills, and the template join the shared-primitive class; (S13) the Tier-2 division of labor is amended per D5's fold.)*
