## §3 — Platform teammate mechanics (official docs; FACT/INFERENCE)

Sources: agent-teams `https://code.claude.com/docs/en/agent-teams`; costs `https://code.claude.com/docs/en/costs`; SendMessage tool doc (this session). **Caching page `https://docs.claude.com/en/docs/build-with-claude/prompt-caching` returned HTTP 403 — not quotable; multipliers below are secondary-sourced and flagged.**

**(a) Context persists & accumulates across SendMessage rounds — FACT, YES.** SendMessage tool doc: *"names keep working after an agent completes (a send resumes it from its transcript)."* agent-teams "Context and communication": *"Each teammate has its own context window … The lead's conversation history does not carry over"* (fresh only at spawn; the teammate's **own** transcript persists thereafter). costs, "Why usage climbs → Long context": *"Claude Code sends your full conversation with every message, so a one-line question in a session that has been open all day uses tokens for the whole conversation, not just the one line."* Each teammate is *"a full, independent Claude Code session"* (agent-teams), so this applies per seat. **INFERENCE: a standing seat re-processes its entire accumulated transcript on every round; cost per round grows with history length.**

**(b) Idle teammate token consumption — PARTIAL / bounded FACT.** costs "Agent team token costs": *"Shut down teammates when their work is done. Each active teammate continues consuming tokens until it exits or the session ends."* costs "Why usage climbs": *"Agent teammates: each active teammate keeps consuming tokens until it exits."* **FACT: keeping a seat alive is billed as a standing liability and the docs advise shutdown. UNKNOWN/not separately quantified: whether a truly idle seat taking zero turns burns tokens continuously vs. only when it next takes a turn — the docs say "active … until it exits" without isolating the zero-turn case.**

**(c) Context window fills → auto-compaction — INFERENCE for teammates; the agent-teams page never mentions compaction.** costs documents it only as a general session behavior: *"auto-compaction, which summarizes conversation history when approaching context limits"*; *"A context or auto-compact warning … The conversation has grown close to the model's maximum input size, and Claude Code summarizes older history to free space"* (lossy); *"Compaction: `/compact` reads the conversation it summarizes, so compacting a large context is itself a large request"* (costs a large read). **INFERENCE: since a teammate is a full session, auto-compaction applies to it. UNKNOWN: teammate-specific compaction behavior/trigger/cost/what it drops is undocumented — the docs never describe compaction inside a teammate.**

**(d) Shutdown / replace / name takeover — FACT, supported.** agent-teams "Shut down teammates": *"The lead sends a shutdown request. The teammate can approve, exiting gracefully, or reject with an explanation."* (Limitation: *"Shutdown can be slow — teammates finish their current request or tool call before shutting down."*) Replacement: troubleshooting, *"Spawn a replacement teammate to continue the work."* Name reuse: SendMessage tool doc — use the raw agentId *"when a newer agent took the name (latest wins)."* **FACT: a seat can be killed (subject to its approval) and a fresh seat can take its name; "latest wins." Caveat (agent-teams Limitations): "No session resumption with in-process teammates — `/resume` and `/rewind` do not restore in-process teammates."**

**(e) Prompt-caching economics for a re-processed long transcript — the re-pay behavior is FACT (official costs doc); the multipliers are secondary-sourced.** costs "Why usage climbs → Cache misses" (official): *"your first message after a break longer than the cache lifetime misses the cache and reprocesses your full context. The lifetime is an hour on a subscription and drops to five minutes once you're drawing on usage credits; on an API key or cloud provider, it's five minutes by default."* **This verifies the lead's hypothesis: a seat idle past the cache lifetime re-pays its full accumulated history at base input price on its next round.** Within the lifetime, caching discounts the re-read. Multipliers (secondary — official page 403, corroborated across caching-pricing write-ups): **cache-read ≈ 0.1× base input; cache-write ≈ 1.25× (5-min TTL) / 2× (1-hour TTL); TTL default 5 min, refreshed on each read** (so continuous chatter keeps the cache warm; a gap longer than the TTL forfeits it). Flag as secondary-provenance; the *behavior* is official.

**Team-vs-single cost, FACT (both official pages):** agent-teams: *"Agent teams … use significantly more tokens than a single session … token usage scales with the number of active teammates."* costs: *"Token usage scales with the number of active teammates and how long each one runs"*; *"Agent teams use approximately 7x more tokens than standard sessions when teammates run in plan mode."* (7× is plan-mode-qualified — not a general multiplier.)

### D1 — Checkpoint-scoped seat recycling in implement (from A1, ruled Q5)

**Statement.** Implement's **producer** (staff-engineer) recycles at cycle checkpoints under a **conditioned cadence** (S5 fold, user-ruled at review): recycle when the transcript has plausibly outgrown a fresh brief **or** the cache has gone cold — approximated by two lead-countable conditions, a **cycle floor (~≥3; the conservative side of the rationale's ~cycle-2 crossover estimate)** and a **gate-pause-duration check** (a pause past the cache lifetime forfeits the discount that made standing cheap); exact values are build-scoped, calibrated by the D4 probe. Early-cycle / warm-cache checkpoints keep the seat standing — unconditional per-checkpoint recycling would *increase* tokens there (DQ6: on a subscription the cache lives an hour, and a minutes-long gate approval leaves it warm while the young transcript still undercuts a fresh brief). The **verifier** (qa-engineer) recycles **per slice boundary**. The dying seat shuts down **when its cycle work is on disk — before the gate pause** (the platform's own "shut down teammates when their work is done"). The successor spawns after gate approval **under the same seat name** (platform name-takeover, "latest wins") — seat identity and roster legibility persist across incarnations. **Retry and fix-pass respawns** (S1/S8 folds): a FAIL verdict's respawn is briefed with the failed-task list **plus the just-failed `cycle-report.md`, relayed at dispatch time** — the lead has just read it to form the verdict and relays it before the next attempt overwrites the file (a later re-read would hit the overwritten report). *Caveat (F-g):* reports overwrite per attempt, so the relay carries the immediately-preceding attempt only, not the full max-3 chain history — a partial substitute for a standing transcript, bounded by the retry cap and per-cycle verification.
