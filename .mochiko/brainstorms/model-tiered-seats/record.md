The session opens from a prior free-form conversation (pasted verbatim by the user at invocation) whose propositions are session **input, not rulings**. Every claim in it — including all numbers — is unverified here until the factual map confirms or refutes it. Seed propositions:

- **P-scribe** — a persistent, pull-based Haiku scribe seat: workers send a pointer + a terse decisions/facts block; the scribe reads artifacts (test output, diffs, templates) from disk itself and writes the templated document. Claimed break-even: savings ≈ (document length − handoff length) at strong-model output rates; negative if the handoff approaches the document. Claimed structural win: format references stop loading into workers; document bodies never transit worker context.
- **P-explorer** — Haiku seats for exploration/audit work, selected by **verification asymmetry**: cheap-to-verify tasks (locate, enumerate, lint) go to Haiku; expensive-to-verify tasks (interpretive reading, design judgment, negative results that drive decisions) stay on strong models. - **P-ladder** — the dispatch ladder: direct tool call → Haiku seat → strong-model seat; each task sits on the lowest rung where the answer can be trusted.
- **P-caution** — never put the discipline surface (spec authoring, slicing, TDD execution under anti-rationalization pressure) on a weak model; where a report is part of the discipline (the executor stating plainly what it did), the worker keeps emitting the terse facts block and only expansion is delegated.

### Addendum — settled fact dispatches (checker-authored, pasted verbatim)

**Fact-check settled — Explore cheap-tier restoration mechanism** *(dispatched during Q5/Q6, settled 2026-07-24)*. Sources: `code.claude.com/docs/en/sub-agents` and `code.claude.com/docs/en/plugins-reference`. Reports what is.

**Sub-fact 1 — can a PLUGIN ship an `Explore` override that shadows the built-in on install? → REFUTED (as an automatic shadow); the documented override path is user/project only, which a plugin cannot occupy.**
- [docs-quoted, plugins-reference §Agents] Plugin agents *"appear in the @-mention typeahead under their **scoped name, such as `my-plugin:code-reviewer`**, once the plugin is enabled"* and *"Plugin agents **work alongside built-in Claude agents**"* — coexist, not replace. Plugin agent frontmatter **does** support `model:` (example shows `model: sonnet`), so a plugin can ship a `model: haiku` agent — but under a scoped identifier.
- [docs-quoted, sub-agents.md "Choose the subagent scope"] The same-name precedence order for file scopes is: **managed (1, highest) > `--agents` CLI (2) > project `.claude/agents/` (3) > user `~/.claude/agents/` (4) > plugin `agents/` (5, lowest)**. Plugin is the lowest file scope.
- [docs-quoted, sub-agents.md] The built-in-Explore override is documented for **"A user or project subagent named `Explore`"** only — plugin scope is not named; and plugin subagents are addressed under their scoped name (`my-plugin:…`), not the bare `Explore` that Claude auto-delegates to.
- **Net:** the doc's cheap-Explore mechanism is a **bare-named `Explore.md` in project (`.claude/agents/`) or user (`~/.claude/agents/`) scope with `model: haiku`** — scopes a plugin does not write into. A plugin can only ship a **distinct scoped seat** (e.g. `mochiko:explorer`, `model: haiku`) that a command dispatches explicitly; it cannot auto-shadow native `Explore`. **[could-not-verify]** only on an explicit "a plugin cannot override the built-in Explore" sentence — but every documented fact (scoped naming · "alongside" built-ins · user/project-only override · lowest priority) points against the shadow.

**Sub-fact 2 — does the per-invocation `model` param apply to the BUILT-IN `Explore` type (spawn "Explore on haiku" with no agent file)? → COULD-NOT-VERIFY (suggestive-yes, doc-unconfirmed for the built-in specifically).**
- [docs-quoted, sub-agents.md "Choose a model"] The general resolution order is stated for *"a subagent"* (not restricted to custom files): *"When Claude invokes a subagent, it can also pass a `model` parameter for that specific invocation. Claude Code resolves … 1. `CLAUDE_CODE_SUBAGENT_MODEL` env, 2. the per-invocation `model` parameter, 3. the … `model` frontmatter, 4. the main conversation's model."*
- [file-verified, this env's Agent tool schema] The spawn-time `model` param is *"Ignored for subagent_type: 'fork'"* — only `fork` is excepted; `Explore` is not listed as an exception, which by omission suggests it accepts the override.
- [docs-quoted, sub-agents.md built-in-Explore tab] **Counterweight:** Explore's model *"inherits from the main conversation, capped at Opus on the Claude API"*, and the doc's stated way to run it cheaper is a **custom `Explore` file** — *"define one with `model: haiku`"* — never a per-invocation param. If a bare per-invocation `model` reliably forced the built-in Explore cheap, the custom-file instruction would be redundant.
- **Net:** the general order + the fork-only exception suggest a per-invocation `model` would apply to a built-in Explore spawn; the docs never confirm it for the built-in type and route cheap-Explore exclusively through a custom file. No direct confirmation.

**One usage fact bearing on the ruling** [file-verified]: mochiko's current dispatch lines route gaps to *"a native `Explore` pass"* with **no explicit model argument** (e.g. `commands/plan.md:124,152`, `commands/tasks.md:114,135`). So on today's wiring the native Explore mochiko reaches inherits the lead's model (Opus-capped) regardless of sub-fact 2's resolution — a cheap explorer requires either a bare-named project/user `Explore.md` override (not plugin-shippable) or a new, explicitly-dispatched scoped seat.

Settled; cite as the Explore-mechanism fact hereafter.
