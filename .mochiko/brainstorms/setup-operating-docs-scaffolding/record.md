## Decisions

**D1 — The operating-docs layer is a governance concern.** `Confident`
The brainstorm session organisation (`.mochiko/brainstorms/` + `index.md`), `BACKLOG.md`, and `ROADMAP.md` belong in the project's *governance* — part of the constitution — not as an unmanaged convention or a bare scaffolding step. Rationale (user's): it keeps knowledge management organized, and governance is what makes the structure *enforced* rather than merely documented. This resolves the opening fork toward a third reading neither offered option captured: not just scaffolding files at setup, and not just wiring setup's own bookkeeping — the layer is constitutionally owned, which subsumes the scaffolding (setup materializes what the constitution mandates). Fact-checker context: today this layer exists only as mochiko-repo dev convention in `CLAUDE.md`, with no carrier in any command (map §c, §d) — D1 mints the framework-level ruling the map flagged as unstated.

**D2 — The layer enters the constitution as an elective module, recommended default-on.** `Confident` *(reversal — see Corrections & Reversals)*
The knowledge-management layer becomes a constitution **module** (alongside `layer-rules`, `release-gates`, `evolution-notes`), surfaced by a new knowledge-management dimension in setup's interrogation, shipped **default-on**: the user must actively decline it, and once adopted it is fully enforced under the Three-Part Rule like any other principle. Not universal core. Rationale: universal core would reintroduce exactly the fixed-baseline imposition the `setup-constitution-flexibility` ruling killed — the decisive case was the PoC tier, where a two-day spike forced to carry roadmap/backlog/index learns to ignore its constitution, corroding governance credibility everywhere else. Electivity handles the PoC without a waiver ceremony; the default-on posture still puts the structure in front of every user. Default-on vs opt-in was itself weighed in-session (opt-in was on the table as an option and rejected: the projects most needing imposed structure are the least likely to ask for it) *(rationale surfaced into the record post-review, F6)*. Fact-checker context: no existing dimension or module covers this (follow-up check #1) — both the dimension and the module are new mints.

**D3 — The module's contents are a fixed bundle, adopted or declined whole.** `Confident`
Adopting the knowledge-management module means adopting the four-part system as one unit: `.mochiko/brainstorms/<slug>/` session organisation + `index.md` carrying the maintenance contract, plus repo-root `BACKLOG.md` and `ROADMAP.md` with their defined roles and circulation (session rulings graduate to ROADMAP's decision log; open threads land in BACKLOG; the index tracks session status). No inner menu, no layout parameterization — layout mirrors mochiko's own dogfooded shape. Rationale: flexibility already lives at the module-adoption level (D2); a second menu inside the module doubles decision surface for marginal gain, the layer's value is the circulation between parts (an index whose rulings can't graduate is a dead ledger), and uniform layout preserves cross-project familiarity. A project for which any part feels heavy should decline the module, not adopt a fragment. Adopted on lead recommendation; ratified explicitly by the user ("i will go with your recommendation"). **Rider:** the module text must explicitly disambiguate its `ROADMAP.md` from the per-project brownfield `evolution-roadmap.md` stub (fact-checker map §b) — same word, different artifact. **Post-review folds:** *(F3)* the seam-level answer to the split alternative: adopting the module does not obligate *running* sessions — the index invariant is vacuously satisfied at zero sessions, so unused session machinery costs one empty `index.md` while BACKLOG/ROADMAP retain standalone value; splitting into sub-modules was judged not worth doubling the adoption surface. *(F5)* containment change acknowledged: adopting this module makes setup write repo-root files for the first time (previously `.mochiko/memory/` only — map §a); accepted deliberately — the operating docs are for humans and repo-root is where humans and renderers look, `.mochiko/` is machinery — with every root write governed by the D6 never-overwrite floor.

**D4 — Enforcement rides three carriers, adopted as a chain.** `Confident`
(1) **Command bookkeeping:** `/mochiko:brainstorm` (and any future session-producing command) carries the index contract as executable steps — read `index.md` before opening, add the entry on open, update status at acceptance/supersession — with a graceful no-module branch for projects that declined the layer. This is a concrete build item: brainstorm.md today has zero index references (map §c). (2) **CLAUDE.md sync:** the module adds rows to the constitution's existing Mandatory Sync Artifacts table so the target project's CLAUDE.md carries the operating-manual text — the mechanism proven by mochiko's own repo; honestly stub-backed until `syncing-claude-md` ports (follow-up check #1). (3) **Testable invariants:** the module's Testability line gets mechanical checks — (a) every directory under `.mochiko/brainstorms/` has an index entry and every index status matches its record's Status line; (b) all four bundle artifacts are present *(F1)*; (c) every index entry marked accepted names where its outcome landed — a ROADMAP row or an explicit no-graduation *(F1)*. Content *quality* of BACKLOG/ROADMAP is explicitly exempt from mechanical enforcement — that boundary is declared, not implied *(F1)*. **Executor named *(F2)*:** session-producing commands run these invariants at session open/close under mochiko's fix-on-sight rule (the same read-the-index-first step carrier 1 already mandates), and the constitution validator re-audits at setup/amend — no standing kernel, keeping the kernel-free constraint intact. Rationale: declaration without carriers is the exact failure the fact-checker mapped (contract in prose, no executor); the three carriers cover write-time, ambient-session, and audit-time failure modes respectively. Adopted on lead recommendation (all three kept); ratified explicitly by the user ("i will go with your recommendation").

**D5 — Elicitation: a new knowledge-management dimension slotted after "existing practices & tools" (agenda becomes ten dimensions, KM at #7); materialization: G5 finalize scaffolds the adopted bundle.** `Confident`
Rationale: everything the adopt/decline call depends on (tier, type, risk, team, detected practices) is already declared once dimension #6 (existing practices & tools) completes; brownfield evidence about existing docs surfaces in dimension #6 and the KM question lands while it's fresh; a decline still memorializes in the exclusions dimension regardless of when it was decided. G5 is the only sensible materialization point — it already reports accepted artifacts and next steps. User tilted A and asked for the lead's read; recommendation matched the tilt. (Late-placement alternative and fold-into-values alternative steelmanned in-conversation and set aside; the fold-into-values scaling concern — agenda growth per module — is tracked in the Open / Deferred section below.) **Rider (offered as veto-able default, unopposed):** when the module is adopted, G5's next-step suggestion includes `/mochiko:brainstorm` alongside `/mochiko:specify` — finalize is where setup teaches the next move, and the layer exists so sessions have somewhere to land.

**D6 — Brownfield collision policy: codify + per-project ruling, on a never-overwrite floor.** `Confident`
Hard floor: setup never overwrites or renames existing content. Default: existing docs whose semantics fit are codified into the module's roles (an existing decision-log-style `ROADMAP.md` becomes the module's ROADMAP). A true semantic collision (name taken, different meaning — e.g. a product roadmap) surfaces in the KM dimension and the user rules on the spot: adopt the existing doc into the module role, or nest the module's artifact under `.mochiko/`. Rationale: collisions are always high-context — per setup v2's interrogation ethos the user rules on evidence rather than the tool guessing silently; the cost is one evidence-backed interrogation beat paid only when a collision exists. *(F4 concession: the original rationale claimed collisions are "rare" — struck; root `ROADMAP.md` is a common existing filename, often a product roadmap, so the beat may fire routinely. The policy stands regardless: silent defaults were rejected on principle, not frequency, and a routinely-firing beat still costs one question asked with evidence on the table.)* This amends D3's "fixed layout" with its necessary brownfield escape hatch (D3 uniformity holds everywhere a name is free). Alternatives set aside: silent fixed fallback (risks two roadmaps nobody chose), clean-names-or-decline (punishes brownfield projects). Adopted on lead recommendation; ratified explicitly by the user ("i will go with your recommendation").

**D7 — Amend runs surface any module the constitution has no recorded ruling on — once, with the answer recorded either way.** `Confident`
When G1 routes an existing-constitution project to amend mode, the interrogation offers every module (KM included) for which `governance-intent.md` records neither an adoption nor a decline; the answer — either way — is recorded there, so no future amend run re-asks. A recorded decline is permanent until the user reopens it. Rationale: without this, a default-on module only ever reaches greenfield projects and the installed base never hears about it; with the once-only guardrail and durable recording, amend keeps its surgical contract and never becomes a recurring pitch. Alternatives set aside: amend-surfaces-nothing (hollows out default-on), amend-re-offers-each-run (turns amend into an upsell). Adopted on lead recommendation; ratified explicitly by the user ("i will go with your recommendation").
